# Genesis CLI

**一个面向真实代码仓库工作的开源 coding CLI，强调计划执行、工具治理与 agent 协作。**

[English version](README.md)

---

## 项目概述

`Genesis CLI` 围绕一个很简单的判断展开：

> 高质量的 coding workflow，既需要稳定的 runtime 主干，也需要清晰的产品层能力。

这个项目关注的是：

- 把“理解 -> 计划 -> 修改 -> 验证”做成一等体验
- 让工具调用、权限决策和执行结果变得显式、可检查
- 在同一条 runtime 主干上支撑多种交互接口

与其把代码工作理解成一轮又一轮松散的 prompt 对话，`Genesis CLI` 更希望把真实仓库操作变成一种结构化、可审阅、可复现的工程流程。

---

## 设计方向

项目当前遵循几条核心原则：

- **一条 runtime 主干**：交互式、纯文本、结构化和嵌入式接口应共享同一套核心执行语义
- **清晰的产品边界**：计划、权限、编排和呈现不应混写成一层
- **操作可见性**：用户应该能知道系统想做什么、被允许了什么、实际发生了什么
- **长期可维护性**：随着能力增长，架构仍然要保持可审阅和可演进

---

## Genesis CLI 想成为什么

`Genesis CLI` 想成为：

- 一个真正面向本地仓库工作的 coding CLI
- 一个以计划驱动执行和受控工具调用为核心的工程助手
- 一套可以支撑多种接口形态的共享 runtime
- 一个即使能力变强，也仍然保持边界清楚的代码库

它并不以逐项复刻某个现有工具为目标。

## 模式

长期目标是一条 runtime 主干支撑多种接口：

- `Interactive`：面向日常交互
- `Print`：面向终端输出与脚本场景
- `JSON`：面向自动化与 CI
- `RPC`：面向 IDE 与宿主进程

这些模式应当在呈现方式上不同，而不是在核心执行语义上分裂。

## 本地开发

常用命令：

```bash
npm run build --workspaces
npm test
npm run chat:live -- --mode print
npm run test:live:pi-mono
```

本地密钥默认不进入版本控制。开发时使用 `.env.local` 保存本地 API key 和端点配置。
