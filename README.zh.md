# claude-code-pi

**一个基于 `pi-mono` 的 coding CLI 项目，产品层重点放在计划执行、工具治理和 agent 协作上。**

[English version](README.md)

---

## 项目概述

`claude-code-pi` 是一个开源 coding CLI 项目，当前定位是：

- 以 [`pi-mono`](https://github.com/badlogic/pi-mono) 作为唯一运行时内核
- 参考 `Claude Code` 所体现出的高质量 coding workflow
- 将运行时能力与产品层能力明确分层

当前仓库仍处于设计与初始化阶段，主实现尚未在本仓库公开提交。因此这份 README 主要用于说明项目方向、范围和架构判断。

---

## 项目目标

这个项目当前围绕三个工作目标展开：

- 复用成熟运行时基础，而不是重新构建第二套 agent loop
- 把计划、权限、subagent 协作等产品层能力设计成显式系统
- 在同一条 runtime 主干上支持多种交互接口

整体判断并不复杂：运行时稳定性和工作流质量都重要，但它们不必落在同一层中实现。`claude-code-pi` 选择由 `pi-mono` 提供运行时基础设施，而把产品行为和集成设计留在本仓库中完成。

---

## 范围

本仓库规划中的范围包括：

- 计划驱动的任务执行
- 显式的权限策略与工具治理
- 结构化的 subagent 任务契约
- 在同一 runtime 上支持 Interactive、Print、JSON、RPC 四种模式

本项目并不以逐项复刻某个上游工具为目标，而是更关注边界清晰、行为可检查、后续可演进。

## 设计原则

### `pi-mono` 是唯一内核

本项目不再额外构建第二套 agent loop。运行时相关的问题，例如 agent runtime、模型接入、会话机制和终端基础能力，尽量交给 `pi-mono` 负责。

### 产品逻辑放在内核之上

计划协议、权限策略、工具目录、subagent 契约以及不同模式下的呈现方式，都属于产品层。这么做的目的，是在不直接侵入上游内部实现的前提下，保留产品演进空间。

### 计划与验证是一等能力

对于真实代码工作流来说，"理解 -> 计划 -> 修改 -> 验证 -> 汇报" 不应只靠提示词维持，而应当成为明确的产品能力。

### 权限策略需要显式表达

工具调用不应完全依赖模型隐式决策。用户应该能够知道发起了什么操作、为什么被允许或拒绝，以及最终产生了什么结果。

### 一个 runtime 支撑多种接口

交互式 TUI、纯文本输出、结构化 JSON、RPC 嵌入，本质上应该共享同一条 runtime 主干，而不是分别复制一套逻辑。

## 架构

当前设计采用三段式分层：

```text
用户 / IDE / 外部进程
        |
CLI 模式：Interactive / Print / JSON / RPC
        |
体验层：TUI / Formatter / RPC Adapter
        |
产品层：Plan Engine / Permission Policy / Tool Catalog / Subagent Orchestrator
        |
面向 pi-mono 的 Facade 层
        |
pi-agent-core / pi-ai / pi-coding-agent / pi-tui
        |
Filesystem / Shell / Git / MCP / LSP
```

这里最重要的约束是：每项能力都应当能归属到明确层级。运行时基础设施、产品编排和用户界面不应混写成一团。

## 规划中的能力

### 计划驱动执行

任务会尽量按可见流程推进：理解、计划、修改、验证、汇报。重点不是展示“思考过程”，而是让中间状态具备可执行、可检查的意义。

### 工具治理

每个工具都应具备四个明确部分：

| 维度 | 作用 |
|------|------|
| `identity` | 说明这是什么工具 |
| `contract` | 说明如何调用、返回什么 |
| `policy` | 说明风险等级、确认规则、并发规则 |
| `executor` | 说明实际如何执行 |

这样可以把权限、审计和执行边界做成可检查的系统行为，而不是隐含逻辑。

### Subagent 协议

复杂任务应当能被拆成结构化子任务，常见字段包括：

- `goal`
- `allowed_paths`
- `verification`
- `stop_conditions`

这样做的重点，是让任务委派有边界、可审阅，尤其是在多个 agent 同时处理一个仓库时。

### 多模式共享同一 runtime

规划中的接口包括：

- `Interactive`：面向日常交互
- `Print`：面向简单命令行输出
- `JSON`：面向自动化和 CI
- `RPC`：面向 IDE 或外部宿主进程

这些模式预计共享同一个 runtime 和事件主干。

## 与上游的关系

[`pi-mono`](https://github.com/badlogic/pi-mono) 在这个项目中被视为上游依赖和运行时基础。

当前的基本约束是：

- 运行时层面的通用能力，尽量保留在上游
- 产品特有的行为，放在本仓库中实现
- 本地集成优先通过 facade 和 adapter 完成，而不是直接绑定上游内部细节

这样做的目的，是在保留升级能力的同时，降低产品层迭代对上游内部结构的耦合。
